# App Store 심사 거절 리스크 리스트
> FindX (com.junsun.treasurehunter) — 작성일: 2026-04-14  
> 시뮬레이터 실행 및 소스코드 직접 분석 기반  
> 상태: 수정 예정

---

## 🔴 HIGH — 거절 가능성 높음

### 1. Apple 로그인 버튼 — 공식 버튼 미사용 (Guideline 4.8)
- **위치**: `src/components/LoginPage.tsx` (Apple 버튼 렌더링 부분)
- **문제**: Apple이 요구하는 공식 `ASAuthorizationAppleIDButton` 대신 커스텀 SVG + HTML `<Button>` 컴포넌트로 구현되어 있음. 버튼 텍스트도 "Apple로 시작하기"로, Apple 가이드라인 표준 문구("Sign in with Apple")와 다름.
- **Apple 가이드라인**: HIG 및 Sign in with Apple 버튼 가이드라인은 공식 버튼 스타일(폰트, 로고, 최소 크기, 패딩) 준수를 요구함.
- **수정 방향**: `@capacitor-community/apple-sign-in`의 공식 버튼 또는 Apple이 제공하는 네이티브 버튼 컴포넌트 사용.

---

### 2. 리뷰어 로그인 "위장" 코드 (Guideline 2.1, 5.6)
- **위치**: `src/components/LoginPage.tsx` 345번째 줄
- **문제**: 코드 주석에 **"3. 일반 이메일 로그인으로 위장한 리뷰어 로그인 영역"** 이라고 명시되어 있음. 심사관이 UI를 보면 일반 이메일 로그인처럼 보이지만, 실제로는 리뷰어 전용 계정으로만 동작하는 숨겨진 로그인 시스템임.
- **리스크**: App Store 심사 기준 2.1(앱 완성도)과 5.6(개발자 행동강령) 위반. Apple은 심사관을 속이거나 실제와 다른 기능을 제공하는 앱을 즉시 거절하며, 반복 시 계정 제재 가능.
- **수정 방향**: 코드 주석에서 "위장" 표현 제거. 심사용 계정 정보는 App Store Connect의 "리뷰 노트" 또는 "데모 계정" 란에 명시하고, UI는 실제 기능처럼 보이도록 정직하게 구현.

---

### 3. 서비스 약관 / 개인정보 처리방침 — 탭 동작 없음 (Guideline 5.1.1)
- **위치**: `src/components/LoginPage.tsx` 388~398번째 줄
- **문제**: 로그인 화면 하단의 "서비스 약관"과 "개인정보 보호정책" 링크가 `<span>` 태그로만 구현되어 있고 `onClick` 핸들러 또는 `navigate()` 호출이 없음. 클릭해도 아무 동작이 없음.
- **Apple 가이드라인**: 소셜 로그인이 포함된 앱은 로그인 전 약관 및 개인정보 처리방침에 접근 가능해야 함.
- **수정 방향**: `onClick={() => navigate('/terms')}` 및 `onClick={() => navigate('/privacy')}` 추가.

## 🟠 MEDIUM — 거절 가능성 있음

### 4. 앱 시작 시 즉시 푸시 알림 권한 요청 (Guideline 5.1.1)
- **위치**: `ios/App/App/AppDelegate.swift` (didFinishLaunchingWithOptions)
- **문제**: 앱이 시작되자마자 `UNUserNotificationCenter.requestAuthorization()`을 호출하여 푸시 알림 권한을 요청함. 사용자가 앱이 무슨 앱인지 인지하기 전에 권한 팝업이 뜸.
- **Apple 가이드라인**: 권한 요청은 해당 기능이 필요한 컨텍스트에서 요청해야 함. 앱 실행 직후 권한 요청은 거절 사유가 됨.
- **수정 방향**: 온보딩 화면 또는 알림 설정 페이지에서 사용자가 원할 때 권한 요청. AppDelegate에서 `requestAuthorization()` 제거 후 `SettingsPage`의 `handlePushToggle` 흐름으로 통합.

---

### 5. Info.plist — "항상" 위치 권한 선언 vs 실제 사용 불일치 (Guideline 5.1.1)
- **위치**: `ios/App/App/Info.plist`
- **문제**: `NSLocationAlwaysAndWhenInUseUsageDescription`과 `NSLocationAlwaysUsageDescription`이 선언되어 있어 "항상 허용" 위치 권한을 요청할 수 있는 상태. 그러나 개인정보처리방침에는 **"앱 사용 중(When In Use)에만 수집"** 이라고 명시되어 있음. 실제 코드에서도 `getCurrentPosition`만 사용하고 있어 Always 권한이 불필요함.
- **리스크**: 실제 사용하지 않는 "항상" 권한을 선언하면 심사에서 사유 소명 요구 또는 거절.
- **수정 방향**: `NSLocationAlwaysAndWhenInUseUsageDescription`, `NSLocationAlwaysUsageDescription` 키 제거. `NSLocationWhenInUseUsageDescription`만 유지.

---

### 6. Apple 로그인 Nonce 백엔드 미검증 가능성 (Guideline 4.8 / 보안)
- **위치**: `src/components/LoginPage.tsx` (Apple 로그인 로직)
- **문제**: 프론트엔드에서 `secureNonce`를 생성하여 Apple에 전달하지만, 백엔드 서버에서 해당 nonce를 검증하는지 불명확. Apple은 ID Token의 nonce 클레임을 서버에서 반드시 검증할 것을 요구함.
- **리스크**: 보안 취약점으로 인한 심사 거절 또는 출시 후 계정 탈취 리스크.
- **수정 방향**: 백엔드에서 Apple ID Token의 `nonce` 클레임을 SHA-256 해시로 검증하는 로직 구현 확인.

---

### 7. 스토어(포인트 상점) — 인앱결제 우회 가능성 검토 필요 (Guideline 3.1.1)
- **위치**: `src/components/StorePage.tsx`
- **문제**: 포인트로 실물/디지털 상품을 구매하는 스토어가 존재함. 포인트가 현금으로 구매 가능하거나 유료 서비스와 연동될 경우, Apple의 IAP(In-App Purchase) 시스템을 우회하는 것으로 간주될 수 있음.
- **수정 방향**: 포인트가 앱 내 활동(습득물 반환 등)으로만 획득 가능하고 현금 결제가 없다면 심사 노트에 명시. 현금 결제가 포함된다면 Apple IAP 연동 필수.

## 🟡 LOW — 개선 권장

### 8. 앱 기본 언어 설정 한국어 (CFBundleDevelopmentRegion: ko)
- **위치**: `ios/App/App/Info.plist`
- **문제**: 앱의 기본 지역이 `ko`로 설정되어 있음. 글로벌 App Store 출시 시 영어 메타데이터(앱 이름, 설명)가 기본으로 필요함.
- **수정 방향**: 한국 App Store 전용 출시라면 문제없음. 글로벌 출시 시 `CFBundleDevelopmentRegion`을 `en`으로 변경하고 다국어 Localizable.strings 추가.

---

### 9. 앱 버전 정보 하드코딩 (Minor)
- **위치**: `src/components/SettingsPage.tsx` (버전 1.0.0 하드코딩)
- **문제**: 앱 버전이 "버전 1.0.0"으로 하드코딩되어 있어 업데이트 시 소스 수정 필요.
- **수정 방향**: Capacitor의 `App.getInfo()` API로 동적으로 버전 가져오기.

---

### 10. 로그인 페이지 로고 이미지 — 외부 URL 의존
- **위치**: `src/components/LoginPage.tsx` (img src 부분)
- **문제**: 로그인 화면 로고가 외부 서버 URL(`treasurehunter.seohamin.com/api/...`)에서 로드됨. 서버가 불안정하거나 느릴 경우 심사관에게 빈 로고로 보일 수 있음.
- **수정 방향**: 앱 번들 내 로컬 이미지 assets으로 교체.
